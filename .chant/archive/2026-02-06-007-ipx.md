# Add Kiro Skills support to chant init

Kiro v0.9.0 shipped Agent Skills support (kirodotdev/Kiro#4753). Instead of only writing `.kiro/rules.md`, `chant init --agent kiro` should create a proper Kiro skill at `.kiro/skills/chant/SKILL.md` with YAML frontmatter (name, description) and markdown instructions.

Skills format:
- `.kiro/skills/chant/SKILL.md` — required, with `name: chant` and `description` in frontmatter
- Optional `scripts/`, `references/`, `assets/` subdirs
- Kiro auto-activates skills when user requests match the description
- Workspace skills (`.kiro/skills/`) override global skills (`~/.kiro/skills/`)

This replaces or supplements the current `.kiro/rules.md` approach. The skill description should mention spec management so Kiro activates it when the user works with chant specs.

Reference: https://kiro.dev/docs/skills/

## Research: Agent Skills Open Standard

Agent Skills is an open standard published by Anthropic at https://agentskills.io/specification. Already adopted by Microsoft, OpenAI, Cursor, GitHub, Kiro, Copilot, and others.

### Spec Summary

Directory structure:
```
skill-name/
├── SKILL.md          # Required
├── scripts/          # Optional executables
├── references/       # Optional docs
└── assets/           # Optional static resources
```

SKILL.md frontmatter:
- `name` (required): lowercase, hyphens, max 64 chars, must match dir name
- `description` (required): max 1024 chars, describes what + when
- `license`, `compatibility`, `metadata`, `allowed-tools` (optional)

Progressive disclosure: metadata (~100 tokens) loaded at startup, full SKILL.md on activation, resources on demand.

### Provider Skills Directories
- Claude Code: `.claude/skills/` (workspace), `~/.claude/skills/` (global)
- Kiro: `.kiro/skills/` (workspace), `~/.kiro/skills/` (global)
- Cursor: `.cursor/skills/` (nightly, TBC)
- VS Code/Copilot: supported

### Implementation Plan

1. Add `skills_dir() -> Option<&str>` to `AgentProvider` enum
2. Create a `chant` skill (`SKILL.md` with frontmatter + chant workflow instructions)
3. `chant init --agent <provider>` writes skill to provider's skills dir
4. Keep existing `config_filename()` for rules/steering (separate concern)
5. Single `templates/skill-chant.md` template shared across all providers (format is identical)

## Output

## Refined Approach

**No `.chant/skills/`** — skills belong in provider directories per the open standard. Chant's prompts (`.chant/prompts/`) serve the agent execution role; skills serve the interactive/discovery role.

### Implementation Plan (updated)

1. Add `skills_dir() -> Option<&str>` to `AgentProvider`:
   - Claude: `.claude/skills/`
   - Kiro: `.kiro/skills/`
   - Cursor: `.cursor/skills/`
   - Generic: None

2. Create `templates/skill-chant.md` — a single SKILL.md template with proper frontmatter:
   ```yaml
   ---
   name: chant
   description: Manage development specs with chant. Use when working with
     specs, acceptance criteria, or the chant workflow.
   metadata:
     author: chant
   ---
   ```

3. `chant init --agent <provider>` writes skill to `{skills_dir}/chant/SKILL.md`

4. Keep existing `config_filename()` for rules/steering (separate concern, always-loaded)

5. Existing template content should be split: project rules stay in rules file, chant workflow instructions move to the skill
