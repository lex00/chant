# Add research workflow guide with ASCII dependency graph

## Problem

Teams need a comprehensive research workflow guide showing how different spec types work together in research scenarios. The guide should:

- **One guide, two paths** - Academic and Developer research scenarios
- **Show overlaps** - Both use same spec types in similar patterns
- **ASCII dependency graph** - Visual showing how spec types coordinate
- **Scenario-based** - Concrete examples like KPI/OKR and TDD guides
- **Multi-page walkthrough** - Phase-by-phase progression

Currently, `docs/guides/research.md` is a single-page reference without concrete workflows or spec type coordination examples.

## Solution

Create multi-page enterprise guide at `docs/guides/enterprise/research-workflow/` with two parallel paths showing overlapping patterns.

### Guide Structure

```
docs/guides/enterprise/research-workflow/
├── README.md                    # Overview, overlaps, ASCII graph, choose path
├── academic/
│   ├── 01-scenario.md          # PhD student studying climate patterns
│   ├── 02-literature-review.md # Research synthesis (informed_by)
│   ├── 03-data-analysis.md     # Research analysis (origin)
│   ├── 04-pipeline.md          # Driver coordinating phases
│   ├── 05-drift-detection.md   # Data/paper drift detection
│   └── artifacts/
└── developer/
    ├── 01-scenario.md          # Senior dev investigating microservices
    ├── 02-investigation.md     # Research on codebase (informed_by)
    ├── 03-documentation.md     # Documentation specs (tracks)
    ├── 04-implementation.md    # Code + driver coordination
    ├── 05-maintenance.md       # Drift detection on docs/metrics
    └── artifacts/
```

### Academic Scenario Details

**Protagonist**: Dr. Sarah Chen, climate science PhD student

**Research Goal**: Analyze 30 years of Arctic temperature data to identify acceleration patterns

**Workflow:**
1. **Literature Review** (Week 1) - Synthesize 25 papers on Arctic warming
2. **Data Analysis** (Week 2-3) - Statistical analysis of temperature datasets
3. **Pipeline** (Week 4) - Coordinate data cleaning → analysis → visualization
4. **Drift Detection** (Ongoing) - New data added monthly, papers published

**Spec Types Demonstrated:**
- `research` with `informed_by:` for paper synthesis
- `research` with `origin:` for data analysis
- `documentation` with `tracks:` for methodology
- `driver` for multi-phase pipeline
- `schedule: monthly` for recurring analysis

### Developer Scenario Details

**Protagonist**: Alex Torres, Staff Engineer at TechCorp

**Research Goal**: Evaluate microservices migration for 80K LOC monolith

**Workflow:**
1. **Investigation** (Week 1) - Analyze codebase coupling and complexity
2. **Documentation** (Week 2) - Document current architecture
3. **Implementation** (Week 3-4) - POC with 3 extracted services
4. **Maintenance** (Ongoing) - Keep architecture docs current as code evolves

**Spec Types Demonstrated:**
- `research` with `informed_by: src/**` for codebase analysis
- `documentation` with `tracks: src/**` for architecture docs
- `code` specs for POC implementation
- `driver` coordinating multi-service extraction
- Drift detection when tracked code changes
